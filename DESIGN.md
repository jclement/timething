# timething — Design & Architecture

## Overview

timething is a single-page app for visually lining up working hours across
time zones. The user picks a home zone (defaulted from the browser), adds
other zones via a typeahead search, and sees a color-coded grid that
highlights when working-hours overlap. A reference date anchors the grid
so daylight-saving transitions don't silently misrepresent the overlap.

The UI is client-only: all settings live in `localStorage`. A Cloudflare
Worker hosts the SPA and exposes one optional API endpoint — `/api/pdf`
— for server-rendered PDFs via Browser Rendering. The Worker path is
feature-detected at runtime so the app works identically on the free
plan (using `window.print()`) and the paid plan (real PDFs).

## Architecture

```
 Browser
 ├── React SPA
 │   ├── TanStack Router (single route today, ready for /settings etc.)
 │   ├── useSettings  ←→  localStorage: timething:settings:v1
 │   └── useTheme     ←→  localStorage: timething:theme
 │
 │  POST /api/pdf (with settings blob)
 ▼
 Cloudflare Worker (worker/index.ts)
 ├── GET /api/health   → {status, version}
 ├── POST /api/pdf     → ASSETS + BROWSER (optional)
 │                       ↳ launches headless Chromium, navigates to /
 │                         with ?print=1&settings=…, exports PDF
 └── * (default)       → ASSETS binding (SPA fallback via wrangler config)
```

## Key Design Decisions

1. **Client-first, server-optional.** The app is useful with nothing
   deployed — open the HTML locally, it works. The Worker's only job
   today is static hosting, with `/api/pdf` ready for when the user
   flips on Browser Rendering.

2. **Reference date anchors DST.** All calculations start from a
   calendar date + hour in the home zone. That pair is converted to a
   concrete UTC instant, which is then projected into every other zone.
   This avoids the classic bug of "8am Calgary" being ambiguous on the
   spring-forward Sunday.

3. **No third-party time zone library.** We lean on `Intl` for offset
   and name lookups. `Intl.supportedValuesOf("timeZone")` gives the
   full IANA list for free. This saves ~100kb of bundle and keeps the
   tzdata fresh with the browser.

4. **Ranked search over a curated city list.** Raw IANA names like
   `America/Argentina/Buenos_Aires` are awful to search. We maintain a
   small curated list (~130 cities, ~30kb minified) with aliases for
   abbreviations and alt-spellings, then fall back to the IANA list
   for obscure matches.

5. **Single settings blob.** All persistent state is one versioned JSON
   object. Migrations can bump the version key without scattering
   logic across multiple keys.

6. **Two export paths, feature-detected.** The frontend calls
   `/api/pdf` optimistically; a 501 response means the Worker doesn't
   have the `BROWSER` binding, and the frontend silently falls back to
   `window.print()`. Users see one button regardless of plan.

7. **Print styles are first-class.** Color-coding, row layout, and the
   DST footnotes section all use `@media print` plus a print-only
   container. The server-rendered PDF uses the exact same CSS — it
   navigates to the SPA with `?print=1` and renders with Chromium.

## Data Model

Everything persists in a single versioned JSON object under
`timething:settings:v1`:

```ts
interface Settings {
  version: 1;
  homeTz: string;                 // IANA id, e.g. "America/Edmonton"
  zones: ZoneConfig[];            // additional zones in display order
  defaultWorkingHours: { start: number; end: number };
  use24h: boolean;
  theme: "system" | "light" | "dark";
}

interface ZoneConfig {
  id: string;                     // stable uuid for React keys
  tz: string;                     // IANA id
  label?: string;                 // override display name
  workingHours?: { start: number; end: number }; // per-zone override
}
```

The theme preference is kept in a separate key
(`timething:theme`) so the FOUC-prevention script in `index.html` can
read it without parsing the full settings blob.

## Time Math

The core primitives live in [src/lib/time.ts](src/lib/time.ts):

- **`zoneOffsetMinutes(instant, tz)`** — uses `Intl.DateTimeFormat` to
  format `instant` as local wall clock in `tz`, then compares to the
  same wall clock treated as UTC. Handles :30 / :45 zones cleanly.
- **`instantFromHomeHour(tz, date, hour)`** — makes a UTC guess, looks
  up the zone's offset at that moment, corrects. Good enough for the
  grid because the offset changes at most twice a year.
- **`computeDayOffset(...)`** — tells a row cell whether the target
  zone lands on the same calendar date as the home, yesterday, or
  tomorrow.
- **`nextDstTransition(tz, from, lookaheadDays)`** — scans forward a
  day at a time until the offset changes, then bisects within that day
  to the minute. Used for the print-only DST footnotes.

## Frontend Patterns

- **Single page, single source of truth.** `Dashboard` owns all
  dashboard state. Children are presentational.
- **Ctrl-scroll-safe.** The hour grid overflows horizontally on narrow
  viewports; the label column is `sticky left-0` so the zone name
  stays readable while scanning times.
- **Color rotation.** Each zone row pulls `var(--zone-N)` for N mod 8,
  giving the printed page a legible per-row accent without running out
  of distinct hues.
- **No context/reducer.** State is simple enough that `useState` +
  prop-drilling through two layers is cheaper to read than wiring a
  store.

## Testing Strategy

- **Unit**: `src/lib/time.test.ts` covers the DST-sensitive math with
  hard-coded dates (winter vs summer LA, Riyadh in July, Kolkata's
  :30 offset). `src/lib/timezones.test.ts` covers search ranking.
- **E2E**: `e2e/dashboard.spec.ts` runs under desktop Chrome and
  iPhone 14 profiles. Asserts the first render, adding a zone via
  search, and persistence across reload.

## Known Limitations & Future Work

- **No drag to reorder.** Zones are added to the end; there's a
  `<GripVertical>` placeholder in `ZoneRow` for a future DnD hookup.
- **Curated city list is small.** ~130 cities. Good enough for most
  searches, but users with obscure needs still get IANA fallback hits.
- **Print-only sections.** The DST footnotes only render on print.
  Could also be exposed as an on-screen panel.
- **No multi-day view.** Single reference date today; a "next 3 days"
  view could help with recurring standups near DST changes.
- **Browser Rendering cost.** Paid-plan dependency for server PDF is
  documented but not automated — the user flips the binding on
  manually when they want it.
